# Container

Container adalah unit isolasi perangkat lunak yang membungkus aplikasi beserta seluruh dependensinya sehingga dapat berjalan persis sama di lingkungan mana pun. Berbeda dengan mesin virtual yang meniru perangkat keras lengkap dan menjalankan kernel sendiri, container berbagi kernel dengan sistem operasi induk. Isolasi di Linux diwujudkan oleh dua fitur kernel: *namespaces* yang membatasi cakupan pandang proses terhadap sistem berkas, jaringan, dan pengenal proses, serta *cgroups* yang mengatur penggunaan CPU, memori, dan I/O. Karena tidak perlu membawa kernel sendiri, ukuran *image* container sangat kecil (umumnya puluhan megabita) dan bisa menyala dalam hitungan detik.

## Image, Container, dan Registry

Tiga komponen pokok dalam ekosistem ini saling terkait erat. *Image* adalah cetak biru statis yang tersusun dari tumpukan beberapa lapisan *read-only*. Setiap lapisan merekam satu instruksi spesifik, misalnya menyalin berkas atau memasang paket. *Container* merupakan wujud hidup dari *image*. Ketika dijalankan, satu lapisan tulis (*read-write*) ditambahkan di atas tumpukan lapisan-lapisan tersebut; semua perubahan yang terjadi selama container beroperasi hanya tersimpan di lapisan tersebut dan akan musnah begitu container dihapus. *Registry* berfungsi sebagai gudang penyimpanan dan distribusi *image*. Docker Hub dan Quay adalah dua contoh *registry* publik yang paling luas digunakan.

## Mengapa Podman?

Podman (*Pod Manager*) adalah mesin container sumber terbuka yang sepenuhnya mematuhi standar OCI. Perintah-perintahnya sangat kompatibel dengan Docker, bahkan banyak pengguna cukup menambahkan alias `alias docker=podman`. Dua keunggulan utama Podman dibandingkan Docker adalah arsitektur *daemonless* (tidak memerlukan proses latar belakang dengan hak istimewa tinggi seperti `dockerd`) dan dukungan mode *rootless* yang memungkinkan pengguna biasa menjalankan container dengan aman. Jika terjadi kebocoran keamanan, penyerang hanya akan mendapatkan akses pengguna biasa di sistem *host*, bukan kendali penuh *root*. Inilah yang menjadikan Podman pilihan tepat bagi pemula.

## Instalasi dan Verifikasi

Pada distribusi Debian atau Ubuntu, pasang Podman dengan satu perintah: `sudo apt update && sudo apt install podman -y`. Setelah instalasi selesai, jalankan `podman version` untuk memverifikasi bahwa semua komponen telah terpasang dengan benar dan siap digunakan.

## Alur Kerja Dasar

Podman menyederhanakan siklus pengembangan aplikasi menjadi tiga langkah utama: mencari dan menjalankan *image* siap pakai, membangun *image* sendiri dengan Containerfile, serta membagikan *image* ke registry.

### Mencari dan Menjalankan Image Siap Pakai
Untuk menemukan *image* yang sudah tersedia, gunakan perintah `podman search docker.io/busybox`, lalu unduh *image* tersebut dengan `podman pull docker.io/library/busybox`. Setelah terunduh, Anda dapat langsung menjalankan container interaktif melalui `podman run -it docker.io/library/busybox`; Anda akan masuk ke sesi terminal di dalam container. Container yang sedang berjalan dapat dipantau menggunakan `podman ps`, dihentikan dengan `podman stop <id_container>`, dan dihapus dengan `podman rm <id_container>`. *Image* yang tidak lagi diperlukan dapat dihapus melalui `podman rmi <id_image>`.

### Membangun Image Sendiri dengan Containerfile
Apabila *image* publik belum memenuhi kebutuhan, Anda bisa membuat *image* kustom menggunakan berkas *Containerfile*. Contoh berikut membangun aplikasi REST API kecil berbahasa Go yang menangani *endpoint* `/tasks` dengan metode GET dan POST. Pertama, tulis kode aplikasi dalam `main.go`:

```go
package main

import (
	"encoding/json"
	"log"
	"net/http"
)

type Task struct {
	ID   int    `json:"id"`
	Name string `json:"name"`
}

var (
	tasks  []Task
	nextID = 1
)

func getTasks(w http.ResponseWriter, _ *http.Request) {
	w.Header().Set("Content-Type", "application/json")
	json.NewEncoder(w).Encode(tasks)
}

func createTask(w http.ResponseWriter, r *http.Request) {
	var task Task
	if err := json.NewDecoder(r.Body).Decode(&task); err != nil {
		http.Error(w, "Invalid JSON", http.StatusBadRequest)
		return
	}
	task.ID = nextID
	nextID++
	tasks = append(tasks, task)
	w.Header().Set("Content-Type", "application/json")
	w.WriteHeader(http.StatusCreated)
	json.NewEncoder(w).Encode(task)
}

func taskHandler(w http.ResponseWriter, r *http.Request) {
	switch r.Method {
	case http.MethodGet:
		getTasks(w, r)
	case http.MethodPost:
		createTask(w, r)
	default:
		http.Error(w, "Method Not Allowed", http.StatusMethodNotAllowed)
	}
}

func main() {
	http.HandleFunc("/tasks", taskHandler)
	log.Println("http://localhost:8080")
	log.Fatal(http.ListenAndServe(":8080", nil))
}
```

Selanjutnya, buat *Containerfile* yang menerapkan strategi *multi-stage build* agar *image* akhir hanya berisi biner aplikasi tanpa sistem operasi tambahan, sehingga ukurannya sangat kecil.

```dockerfile
FROM golang:1.26-alpine AS builder
WORKDIR /app
COPY go.mod ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o tasks .

FROM scratch
WORKDIR /app
COPY --from=builder /app/tasks .
EXPOSE 8080
CMD ["./tasks"]
```

Pada tahap pertama, kita menggunakan *image* `golang:1.26-alpine` sebagai lingkungan build. Setelah menetapkan direktori kerja, berkas `go.mod` disalin lebih dulu agar proses pengunduhan dependensi bisa di-cache oleh Podman, baru seluruh kode sumber disalin dan dikompilasi menjadi biner statis bernama `tasks`. Tahap kedua dimulai dari *image* kosong `scratch` yang sama sekali tidak membawa sistem operasi atau shell. Biner hasil kompilasi disalin dari tahap sebelumnya ke direktori `/app`, port `8080` dideklarasikan sebagai metadata, dan perintah `CMD` akan langsung menjalankan aplikasi saat container dimulai.

Bangun *image* dengan perintah `podman build -t tasks .` di direktori proyek. Setelah berhasil, *image* lokal `localhost/tasks` siap digunakan. Jalankan container di latar belakang melalui `podman run -d -p 8080:8080 tasks`, lalu uji API dengan `curl http://localhost:8080/tasks`.

### Membagikan Image ke Registry
Untuk mendistribusikan *image* kepada orang lain, unggah ke *registry*. Masuklah ke akun Docker Hub dengan `podman login docker.io`. Beri *tag* baru yang merujuk ke repositori Anda, misalnya `podman tag localhost/tasks docker.io/username/tasks`, kemudian unggah dengan `podman push docker.io/username/tasks`. Pengguna lain kini dapat menarik *image* tersebut menggunakan `podman pull docker.io/username/tasks`.

## Orkestrasi Sederhana dengan Podman Compose

Setelah mampu membangun dan membagikan *image*, sering kali kita perlu menjalankan beberapa container sekaligus sebagai satu kesatuan, misalnya aplikasi backend dengan basis data atau *reverse proxy*. Podman menyediakan perintah bawaan `podman compose` yang akan memanggil penyedia compose yang terpasang di sistem, seperti `podman-compose` atau `docker-compose`. Pastikan salah satunya sudah tersedia—jika belum, pasang dengan `sudo apt install podman-compose`.

Contoh berikut menggabungkan layanan backend Go yang telah kita bangun dengan *reverse proxy* Caddy. Caddy akan menjadi pintu masuk tunggal dari *host*, meneruskan permintaan ke backend di dalam jaringan internal container.

Buat berkas `Caddyfile` dengan isi:

```
:80 {
    reverse_proxy backend:8080
}
```

Kemudian buat `docker-compose.yml`:

```yaml
services:
  backend:
    build: .
    expose:
      - "8080"

  caddy:
    image: caddy:alpine
    ports:
      - "8081:80"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
    depends_on:
      - backend
```

Berkas ini mendefinisikan dua layanan: `backend` yang dibangun dari Containerfile di direktori proyek, dan `caddy` yang ditarik dari *image* `caddy:alpine`. Layanan backend hanya membuka port `8080` di dalam jaringan virtual yang dibuat oleh Compose, sehingga tidak dapat diakses langsung dari *host*. Sebaliknya, container Caddy memetakan port `8081` *host* ke port `80` di dalam container—pemetaan di atas port `1024` diperlukan karena mode *rootless* Podman tidak mengizinkan *binding* ke port sistem. Konfigurasi Caddy dipasang melalui volume `./Caddyfile` dengan opsi *read-only* (`:ro`) demi keamanan, dan `depends_on` memastikan backend sudah berjalan lebih dulu sebelum Caddy dijalankan.

Jalankan seluruh tumpukan dengan `podman compose up -d`. Caddy akan mendengarkan pada port `8081` *host* dan meneruskan setiap permintaan ke container backend di port `8080`. Akses aplikasi melalui `http://localhost:8081/tasks`. Untuk menghentikan dan membersihkan semua layanan, gunakan `podman compose down`.

## Penanganan Masalah dan Perawatan Rutin

Jika container gagal berjalan, periksa *log* terlebih dahulu menggunakan `podman logs <nama_container>`. Informasi yang tercetak di *log* hampir selalu menunjukkan penyebab pasti kegagalan. Apabila belum jelas, lakukan inspeksi mendetail dengan `podman inspect <nama_container>` dan perhatikan *exit code* serta variabel lingkungan yang digunakan. Untuk penyelidikan langsung, Anda dapat menjalankan container dari *image* yang sama secara interaktif dengan *shell* diagnostik: `podman run -it --entrypoint /bin/sh <nama_image>`.

Lakukan pembersihan berkala menggunakan `podman system prune -a`. Perintah ini akan menghapus semua *image*, container, dan *volume* yang sudah tidak terpakai, menjaga lingkungan pengembangan tetap ringan dan responsif.

Dengan menyelesaikan seluruh langkah di atas, Anda telah menguasai alur kerja dasar container menggunakan Podman—mulai dari menarik *image*, membangun aplikasi sendiri, mendistribusikannya, hingga mengorkestrasi layanan dengan Compose. Selanjutnya, Anda dapat memperdalam topik seperti manajemen volume agar data tetap bertahan, konfigurasi jaringan kustom, atau pengoptimalan Containerfile agar proses build semakin efisien.
